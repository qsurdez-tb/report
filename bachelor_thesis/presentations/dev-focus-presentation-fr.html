État actuel

  • Pipeline CI/CD cassé, aucun nouveau développement possible
  • Python 2.7 en fin de vie : dette technique et de sécurité, librairies avec CVEs connus
  • Images servies sans watermark ni stéganographie, faciles à copier
  • Le watermark existant est facilement contournable, peu résistant aux transformations d'image
  • Code mort : rôles inutilisés, intégration PiAnoS non fonctionnelle
  • La couche de cryptographie est obscure, difficile à analyser et utilise un ancien algorithme (AES-CBC)
  • Ergonomie insuffisante, l'application n'est pas activement utilisée par les chercheurs aujourd'hui

1. Déploiement stable ~3-4 jours

  • Redéployer sur une nouvelle machine gérée par le service informatique de l'ESC Remplacer Docker Swarm par un simple docker compose
  • Configurer le reverse proxy et importer la base de données de production Via Caddy et la sauvegarde de la base de données effectuée le 19.05, ~2 heures
  • Stocker les dépendances homemade (MDmisc, NIST…) dans le GitLab de l'ESC Actuellement disponibles sur une seule machine, ~1-2 heures
  • Mettre à jour Python vers la version 3.9+ pour accéder à des librairies modernes et sans CVE Tâche la plus importante car elle s'étend aux dépendances NIST, MDmisc, etc.

2. Watermarking par downloader ~5 jours

  • Envoyer un lien pour télécharger le zip aux downloaders avec code
  • Intégrer une marque unique par utilisateur dans chaque image téléchargée
  • Maintenir un historique des téléchargements : qui a téléchargé quelles images et quand
  • Encoder un code secret dans le watermark, lié à l'adresse e-mail du demandeur Permet la traçabilité si une image divulguée est retrouvée

3. Logs d'accès aux images ~5 jours

  • Enregistrer chaque accès : ID utilisateur + ID objet + horodatage
  • Fournit une piste d'audit pour toutes les requêtes d'images
  • Périmètre conséquent

4. Stéganographie sur toutes les images servies ~10 jours

  • Étudier l'état de l'art en stéganographie robuste aux transformations d'image Revue de littérature, définition du périmètre avec le superviseur
  • Intégrer une trace cachée par utilisateur dans chaque image servie
  • Objectif : une traçabilité qui résiste au recadrage, à la compression et à la conversion de format ~10 jours pour la recherche et l'implémentation

5. Expérience utilisateur ~7 jours

  • Priorité au rôle Trainer, le plus activement utilisé aujourd'hui Retirer une image d'un dossier, supprimer un dossier, ~2 jours
  • Filtrage par qualité Détection automatique de la qualité + recherche/filtrage par qualité d'impression, ~5 jours
  • Objectif : rendre la plateforme suffisamment utilisable pour que les chercheurs l'adoptent réellement
